# 0.3 Operator / Deploy Split

0.3 separates host convergence from app deployment. The old 0.2 shape used one
`admin` account for both jobs: Ansible connected as `admin` with broad sudo, and
the app CLI also connected as `admin` to run the narrow privileged helper. That
worked, but it meant CI deploy credentials inherited operator-grade access.

The 0.3 target has three identities:

```text
bootstrap user   root or provider-created initial account; install phase 1 only
operator user    default operator; broad sudo for Ansible and recovery
deploy user      default deploy; narrow sudo for /usr/local/bin/simple-vps only
```

Fresh 0.3 installs create both `operator` and `deploy`. Ansible phase 2 connects
as `operator` and uses `/etc/sudoers.d/operator`:

```text
operator ALL=(ALL) NOPASSWD:ALL
```

App commands use the manifest `server` target, which should be `deploy@host`.
The deploy account gets only `/etc/sudoers.d/simple-vps`:

```text
deploy ALL=(root) NOPASSWD: /usr/local/bin/simple-vps
```

`/etc/sudoers.d/admin` is not part of the 0.3 fresh-host contract.

## SSH Keys

The security boundary is the Unix account plus sudoers grant, not a hidden
server-side policy in the laptop CLI. The installer supports one operator public
key and one deploy public key. Fresh installs require a deploy key unless
`--shared-key` is passed explicitly. Shared-key installs are allowed for small
single-person setups, but they are visibly weaker because the same laptop key can
enter both the broad sudo and narrow deploy identities. Team and CI docs should
recommend a deploy-only key for CI and a separate human/operator key for host
recovery.

The committed manifest should point at the deploy identity:

```toml
[env.production]
server = "deploy@100.x.y.z"
```

The operator identity is not committed to app repos. It belongs to install and
host maintenance workflows.

## Install Flow

Remote install still starts from a bootstrap SSH user, usually `root`. Phase 1
creates the operator and deploy users, installs their authorized keys, and
writes the two sudoers files. Phase 2 verifies operator SSH, then runs Ansible as
`operator` with `become: true`.

Local install can perform both phases in one process because it is already
running on the box. It still writes the same users, keys, and sudoers files.

The old `--admin-user` flag should remain as a compatibility alias during 0.3,
but new docs and prompts should prefer `--operator-user` and `--deploy-user`.

## Doctor Contract

`simple-vps host doctor` reports the split as healthy only when:

- operator and deploy resolve to different users.
- `/etc/sudoers.d/operator` grants broad sudo to the operator user.
- `/etc/sudoers.d/simple-vps` grants only `/usr/local/bin/simple-vps` to deploy.
- deploy does not have broad passwordless sudo.
- the legacy `admin` conflation is not present.

The legacy 0.2 shape is degraded, not fatal: `admin` with broad sudo plus the
narrow helper grant means the host can still deploy, but it does not match the
0.3 security model.

## Existing 0.2 Hosts

There is no automatic in-place migration in 0.3. Existing hosts keep working,
but doctor reports degraded until the operator/deploy split is applied manually.

Manual migration is:

1. SSH as the existing `admin` or root recovery user.
2. Create `operator` and `deploy`.
3. Install the operator and deploy authorized keys.
4. Write `/etc/sudoers.d/operator` and `/etc/sudoers.d/simple-vps`, then validate
   both with `visudo -cf`.
5. Verify `ssh deploy@host 'sudo -n /usr/local/bin/simple-vps status'`.
6. Update app manifests from `admin@host` to `deploy@host`.
7. Run a deploy.
8. Remove `/etc/sudoers.d/admin` after the deploy path is proven.
9. Run `simple-vps host doctor`.

This is deliberately manual because deleting the old broad admin path before the
new deploy path is proven is a lockout risk.
